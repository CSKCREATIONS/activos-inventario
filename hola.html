1. Resumen ejecutivo

El sistema tiene una base funcional avanzada (autenticación, activos, usuarios, asignaciones, documentos, suministros, licencias, dashboard y generación PDF). Sin embargo, presenta errores críticos que impiden el arranque estable o generan riesgos de seguridad. Antes de añadir nuevas funcionalidades, se debe ejecutar una fase de estabilización inmediata.

  • ❌ El backend no arranca si falta JWT_SECRET en .env.
  • ❌ models/susuario.py contiene código de rutas en lugar de modelo → errores de importación.
  • ❌ La página de reportes (ReportesPage) exporta un hook, no un componente → falla de compilación.
  • ❌ Documentos PDF en /uploads/ están expuestos públicamente sin autenticación.
  • ❌ El acta de entrega puede descargarse como plantilla en blanco (sin datos de la asignación).
  • ⚠️ Migraciones desordenadas, falta de normalización en usuarios múltiples, columna sede ausente.

Una vez corregidos estos bloqueos, se recomienda seguir el roadmap por fases (seguridad, base de datos, flujo documental y funciones de valor).

3. Problemas críticos de corrección inmediata (orden sugerido)

3.1. Backend no arranca por falta de JWT_SECRET

Archivo: routes/auth.py. Se exige la variable pero no está en .env.
Solución: Agregar JWT_SECRET=clave_segura_larga en backend_py/.env y excluir .env del repositorio.

3.2. models/susuario.py mal estructurado

Contiene APIRouter y lógica de rutas. Debe contener solo funciones de acceso a BD.
Solución: Mover rutas a routes/susuarios.py y en models/susuario.py dejar: get_by_username, create_usuario_sistema, count_usuarios_sistema, update_ultimo_acceso.

3.3. Falta la dependencia pypdf

requirements.txt no incluye pypdf pero acta_entrega_pdf.py lo importa.
Solución: Añadir pypdf==5.1.0 (o PyPDF2) y ejecutar pip install -r requirements.txt.

3.4. Acta de entrega puede descargarse sin datos (plantilla en blanco)

Archivo: routes/asignaciones.py. Si detecta la plantilla, la devuelve tal cual.
Solución: Forzar siempre generar_acta_entrega_pdf() para rellenar con los datos de la asignación.

3.5. Script regenerate_acta.py con parámetro inexistente

rellenar=True no existe; debe ser regenerar=True.
Solución: Cambiar el argumento en scripts/regenerate_acta.py.

3.6. Página de reportes rota (exporta hook en lugar de componente)

views/pages/reportes/ReportesPage.tsx contiene el controlador.
Solución: Reconstruir la vista que importe useReportesController desde el controlador.

3.7. Inconsistencia de estado: "Desposicion" vs "Disposición"

La migración usa 'Desposicion', el frontend usa 'Disposicion'.
Solución: Unificar a 'Disposición' en BD, tipos TypeScript y formularios.

5. Base de datos y migraciones

5.1. Problemas identificados

  • No existe migración base que cree todas las tablas principales.
  • Las migraciones no tienen numeración secuencial → run_migrations.py ejecuta en orden alfabético, riesgo de fallo.
  • Columna sede en usuarios se maneja con try/except pero nunca se agregó formalmente.
  • usuarios_ids como LONGTEXT JSON en asignaciones → dificulta consultas y auditoría.

5.2. Soluciones recomendadas

  • Crear 001_create_base_tables.sql con todas las tablas (equipos, usuarios, asignaciones, accesorios, documentos, etc.).
  • Renombrar migraciones existentes con prefijos 002_*, 003_* según orden lógico.
  • Ejecutar migración específica para agregar sede: ALTER TABLE usuarios ADD COLUMN sede VARCHAR(150) DEFAULT NULL;
  • Crear tabla puente asignacion_usuarios en lugar del campo JSON.

6. Mejoras por módulo (adicionales a correcciones)

6.1. Activos

  • Separar estado operativo de es_rentado.
  • Historial de cambios del activo (tabla historial_equipos).
  • Proceso de baja controlada con motivo, fecha, responsable y documento soporte.
  • Campos de garantía y generación de código QR por placa.

6.2. Asignaciones

  • Flujo formal: crear asignación → generar acta → firmar acta (digital) → registrar devolución.
  • Sincronizar estado de accesorios al editar asignación (no solo al crear/devolver).
  • Generar acta de devolución con estado físico y accesorios faltantes.

6.3. Documentos

  • Previsualización de PDF.
  • Estados del documento: pendiente, generado, firmado, validado, vencido.
  • Control de versiones estricto.

6.4. Suministros y licencias

  • Kardex de movimientos (entradas/salidas) para suministros físicos.
  • Alertas de stock bajo.
  • Vista independiente de licencias de software con control de vencimiento.

6.5. Dashboard y reportes

  • Centralizar KPIs en backend.
  • Exportar a CSV, Excel y PDF con filtros previos.
  • Reportes predefinidos: inventario, activos asignados, equipos sin documentos, mantenimientos, bajas, etc.
